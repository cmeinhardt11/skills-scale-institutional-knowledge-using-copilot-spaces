# OctoAcme Project Management Docs

This README serves as the central entry point for accessing OctoAcme's standard project management processes, key roles, and workflows. Use the links below to navigate to the relevant process documents and find detailed guidance on how we plan, execute, and continuously improve our projects.

## Overview of Project Management Processes

### Principles

OctoAcme runs projects according to five core principles:

- **Customer-first**: Prioritize customer value and usability in every decision.
- **Iterative delivery**: Ship in small, testable increments to reduce risk and gather feedback early.
- **Clear ownership**: Every project has a named Project Manager and Product Lead who are accountable for outcomes.
- **Data-informed decisions**: Measure impact, analyze results, and iterate based on evidence.
- **Psychological safety**: Encourage open feedback, experimentation, and learning from failures.

### Core Roles

| Role | Responsibilities |
|------|-----------------|
| **Project Manager** | Coordinates delivery, manages schedules and risk, owns stakeholder communications. |
| **Product Manager** | Defines outcomes, prioritizes the backlog, and measures success. |
| **Developers** | Implement features and collaborate on design and technical decisions. |
| **QA** | Validates quality and verifies acceptance criteria are met. |
| **Stakeholders** | Provide inputs, review progress, and give approvals. |

### Project Lifecycle

Projects move through five phases:

1. **Initiation** — Define the problem statement, identify stakeholders, and establish a high-level timeline.
2. **Planning** — Finalize scope, allocate resources, set milestones, and map dependencies.
3. **Execution** — Build, test, review, and iterate in short cycles.
4. **Release** — Deploy to production, verify functionality, and announce to stakeholders.
5. **Close & Retrospective** — Capture learnings, document outcomes, and close out the project.

### Key Artifacts

- Project Charter / One-pager
- Roadmap and Release Plan
- Sprint / Iteration Backlog
- Acceptance Criteria & Definition of Done
- Risk Register
- Retrospective notes and action items

### Communication Cadence

| Cadence | Meeting |
|---------|---------|
| Weekly | PM + PdM sync |
| Twice-weekly | Standups |
| Monthly | Stakeholder updates |
| As needed | Ad-hoc escalations |

### Risk Management

Risks follow an **Identify → Assess → Mitigate → Monitor** cycle. Escalation paths move from team-level → PM → Product Lead → Sponsor.

### Quality & Testing

All changes are validated through:

- Unit tests, integration tests, and end-to-end smoke tests
- Security scanning in CI
- Manual QA when automated coverage is insufficient

### Continuous Improvement

Retrospectives are held after sprints, releases, and incidents. Action items are tracked with owners and due dates to ensure follow-through.
